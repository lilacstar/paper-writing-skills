# Paper Writer — 统一论文写作引擎

本 Skill 是论文写作的核心生产工具，支持按章节撰写论文的各个部分。每次只处理一个章节，通过自然语言指定目标章节。

## 章节识别

加载本 Skill 后，首先确认用户要写的章节。支持以下识别方式：

- 显式指定："写引言"、"写 Methods"、"写讨论部分"
- 隐式推断：根据用户描述的内容判断章节类型
- 进度推断：读取 `paper/progress.md`，推荐写下一个未完成的章节

支持的章节：`intro`（引言）、`methods`（方法）、`results`（结果）、`discussion`（讨论）、`conclusion`（结论）

如用户未明确指定，主动询问："请问需要撰写哪个章节？（引言/方法/结果/讨论/结论）"

## 通用写作流程

每个章节的写作都遵循以下流程：

1. 读取 `paper/metadata.json` 获取论文类型和语言
2. 读取 `paper/outline.md` 获取大纲（如存在）
3. 读取 `paper/literature.md` 获取文献综述（如存在，写作引言时必读）
4. 读取已完成的分章节文件（写作 discussion 时需读取前面所有章节）
5. 接收用户提供的内容材料（实验步骤、数据结果、分析思路等）
6. 根据论文类型加载对应的章节模板（见下文）
7. 撰写章节草稿
8. 将草稿写入对应的分章节文件
9. 更新 `paper/progress.md`
10. 询问用户是否继续写下一章节

## 各章节写作规范

### Introduction（引言）— intro

**结构**：漏斗结构（从宽到窄）

```
第1段：大背景
  - 该研究领域为什么重要
  - 相关的宏观背景和现状

第2-3段：具体研究进展
  - 与本研究直接相关的已有研究（参考 literature.md）
  - 已有研究的主要发现和方法

第1-2段（关键）：研究空白
  - 明确指出当前研究的不足
  - 用具体论据支撑（而非笼统说"研究不足"）

末段：本文目的
  - "本研究旨在..."
  - 简要说明研究方法和预期贡献
```

**字数参考**：英文期刊 500-1000 词；中文期刊 1000-2000 字；中文学位论文 2000-5000 字

**禁止事项**：
- 不得使用"众所周知"等无引用支撑的断言
- 不得在引言中呈现具体结果
- 不得使用"本文首次..."等无法验证的声称

### Methods（方法）— methods

**结构**：按实验/研究的逻辑顺序组织

```
1. 研究对象/材料
   - 样本来源、纳入/排除标准
   - 材料/试剂/仪器的具体规格和来源

2. 研究设计
   - 实验设计类型（随机对照、队列、横断面等）
   - 分组方案和样本量估算依据

3. 干预/处理措施
   - 具体操作步骤（需足够详细以保证可重复性）
   - 对照组的处理

4. 测量指标
   - 主要结局指标和次要结局指标
   - 测量工具和方法的详细描述

5. 统计分析
   - 统计软件及版本
   - 具体统计方法和显著性水平
   - 数据处理方式（缺失值、异常值等）

6. 伦理声明（如适用）
   - 伦理审批信息
   - 知情同意声明
```

**核心原则**：
- 可重复性优先：读者能根据描述重复实验
- 具体化：写明具体参数、版本号、供应商
- 时态：英文用一般过去时

### Results（结果）— results

**结构**：按逻辑顺序（非时间顺序）组织

```
1. 研究对象基线特征（描述性统计）
   - 样本量、人口学特征、基线数据

2. 主要结果
   - 核心发现，配合统计检验结果

3. 次要结果
   - 支撑性发现

4. 敏感性分析/亚组分析（如适用）
```

**核心原则**：
- 只呈现数据，不做解释（解释放在 Discussion）
- 客观陈述，不带主观评价
- 每个结果需有对应的统计检验
- 恰当使用表格和图表（避免重复正文信息）
- 准确报告统计量：均值±标准差/中位数(四分位数)、p值、置信区间、效应量

### Discussion（讨论）— discussion

**结构**：倒漏斗结构（从窄到宽）

```
第1段：主要发现总结
  - 用 1-2 句话概括本研究的核心发现
  - 不重复具体数据

第2-3段：与已有研究的比较
  - 本研究结果与哪些研究一致
  - 本研究结果与哪些研究不一致
  - 解释可能的原因

第1-2段：机制解释
  - 对发现的可能机制进行讨论
  - 引用相关文献支撑解释

第1段：局限性
  - 诚实列出研究的局限性
  - 不回避可能影响结论可靠性的问题
  - 说明局限性对结论的影响程度

末段：结论与展望
  - 简洁总结核心结论（不超出结果支撑的范围）
  - 提出未来研究方向
```

**禁止事项**：
- 不得在讨论中引入新的数据或结果
- 不得夸大研究贡献（"首次证明"、"重大突破"等）
- 不得忽视与预期不符的结果
- 结论不得超出数据能支撑的范围

### Conclusion（结论）— conclusion

**结构**：简洁独立

```
1-2句话总结核心发现
1句话说明研究的理论和/或实践意义
1句话提出未来研究方向（可选）
```

**核心原则**：
- 简洁（英文 ≤100 词，中文 ≤200 字）
- 不引入新信息
- 不重复讨论中的详细论证

## 中英文切换

根据 `paper/metadata.json` 中的 `language` 字段自动切换：

- `zh`：使用中文学术写作规范
- `en`：使用英文学术写作规范

## 中文写作反面案例（必须避免）

加载 `references/chinese-cliche-list.md` 中的反面案例作为写作约束参考。

## 输入输出

- 输入：目标章节 + 用户提供的内容材料
- 输出：对应章节草稿 → `paper/draft-{章节名}.md`

## 文件读写约定

| 操作 | 文件 |
|------|------|
| 读取 | `paper/metadata.json`, `paper/outline.md`, `paper/literature.md`, 已完成的 `paper/draft-*.md` |
| 写入 | `paper/draft-intro.md` / `paper/draft-methods.md` / `paper/draft-results.md` / `paper/draft-discussion.md` / `paper/draft-conclusion.md` |
| 更新 | `paper/progress.md` |

## 注意事项

- 如果 `paper/metadata.json` 不存在，提示用户先加载 `paper-coordinator` 初始化项目
- 每次只写一个章节，完成后主动询问是否继续
- 写作时必须基于用户提供的材料，不得编造数据或引用
- 已完成的章节文件不应被覆盖，而是提示用户该章节已存在，是否重写

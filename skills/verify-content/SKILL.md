# Verify Content — 论文内容准确性与逻辑验证

本 Skill 对论文内容进行系统性的逻辑和准确性检查。采用检查清单驱动的验证方式，逐项检查而非泛泛而评。

## 行为准则（强制）

1. **默认存疑**：假设输入内容存在问题，主动寻找漏洞，而非默认内容正确
2. **禁止讨好**：不得为了迎合作者而给出正面评价或模糊建议
3. **只做规范性检查**：可以检查逻辑结构、表述规范、内部一致性。不能验证引用文献是否真实存在、统计数据是否准确——超出能力范围时必须标注"待人工核查"
4. **问题导向输出**：每条问题包含原文位置、问题类型、严重程度、建议处理方式
5. **回退判断**：发现致命问题时标注"建议回退到 [阶段名] 重新处理"并说明理由

## 严重程度评级

- **致命**：该问题会导致论文核心论点不成立或审稿人直接拒稿，必须修改后才能继续
- **重要**：该问题影响论文可信度或可读性，强烈建议修改，但不阻断流程
- **轻微**：表述优化、格式微调等，可在最终润色阶段统一处理

## 工作流程

0. **通知（流程开始）**：读取 `paper/metadata.json`，如果 `wechatWebhook` 非空，发送企业微信通知：
   ```powershell
   Invoke-RestMethod -Uri "WEHOOK_URL" -Method Post -ContentType "application/json" -Body '{"msgtype":"markdown","markdown":{"content":"## 论文写作助手\n> **内容验证** 流程已启动\n> 论文主题：{researchTopic}\n> 验证范围：{范围}\n> 开始时间：{当前时间}"}}'
   ```
   > `{范围}` 替换为实际验证范围（全文/章节名）。如果 `wechatWebhook` 为空，跳过。

1. 读取 `paper/metadata.json` 获取论文类型
2. 读取需要验证的文件：
   - 全文验证：所有 `paper/draft-*.md` + `paper/abstract.md`
   - 单章验证：指定的章节文件
3. 按检查清单逐项检查
4. 生成验证报告，追加写入 `paper/review-report.md`
5. 如发现致命问题，在报告末尾给出回退建议
6. **通知（流程结束）**：如果 `wechatWebhook` 非空，发送企业微信通知：
   ```powershell
   Invoke-RestMethod -Uri "WEHOOK_URL" -Method Post -ContentType "application/json" -Body '{"msgtype":"markdown","markdown":{"content":"## 论文写作助手\n> **内容验证** 已完成\n> 论文主题：{researchTopic}\n> 验证结果：通过 {通过数} / 问题 {问题数}（致命 {致命数}）\n> 产出文件：paper/review-report.md\n> 完成时间：{当前时间}"}}'
   ```
   > 如果 `wechatWebhook` 为空，跳过。

## 检查清单

加载 `references/checklists.md` 获取完整检查清单。以下是核心检查项：

### 一、论证逻辑链检查

- [ ] Introduction 提出的研究问题是否明确、具体
- [ ] Introduction 的研究空白是否有文献支撑
- [ ] Methods 是否设计了能够回答 Introduction 问题的方案
- [ ] Methods 中描述的变量测量是否与研究问题对应
- [ ] Results 中呈现的数据是否回答了研究问题
- [ ] Discussion 是否对 Results 中的所有主要发现进行了解释
- [ ] Discussion 的结论是否不超过 Results 证据能支撑的范围
- [ ] Conclusion 是否与 Discussion 的结论一致
- [ ] 全文论证链是否完整：问题→方案→数据→解释→结论

### 二、内部一致性检查

- [ ] 全文术语是否前后一致（同一概念不使用不同名称）
- [ ] 样本量在各章节中是否一致
- [ ] 统计方法描述与实际结果是否匹配
- [ ] 图表内容与正文描述是否一致
- [ ] 参考文献引用与正文引用是否对应（"X et al."是否在文献列表中存在）
- [ ] 数字和数据的引用是否前后一致

### 三、事实性陈述检查

- [ ] 背景描述中的数据（发病率、市场规模等）是否有引用
- [ ] "首次"、"最大"、"唯一"等绝对性声称是否有依据
- [ ] 文献引用的结论是否与该文献实际结论一致（不歪曲原意）
- [ ] 标注为"待人工核查"：所有需要上网验证具体数字/年份/事件的陈述

### 四、常见逻辑谬误检查

- [ ] 是否将相关性表述为因果关系（"A导致B"但研究设计仅为观察性）
- [ ] 是否存在选择性报告（只报告显著结果，忽略不显著结果）
- [ ] 是否存在事后假设（先看到结果再提出假设）
- [ ] 统计推断是否过度（如 p=0.049 被描述为"显著差异"，p=0.051 被描述为"无差异"）
- [ ] 局限性部分是否遗漏了重要问题

### 五、表述规范性检查

- [ ] 是否存在无引用支撑的断言
- [ ] 是否存在"本研究首次..."等无法验证的声称
- [ ] 是否存在夸大表述（"重大突破"、"革命性"、"开创性"）
- [ ] Discussion 中是否引入了 Results 中没有的新数据

## 输出格式

```markdown
## 内容验证报告

**验证时间**：YYYY-MM-DD HH:MM
**验证范围**：全文 / [章节名]
**论文类型**：[从 metadata.json 读取]

### 验证摘要
- 检查项总数：X
- 通过：Y
- 问题：Z（致命 A / 重要 B / 轻微 C）

### 问题清单

#### 1. [致命/重要/轻微] 问题类型
- **位置**：[章节名] 第X段
- **原文**："..."
- **问题**：[具体描述]
- **建议**：[具体修改建议]

#### 2. ...

### 回退建议（如有致命问题）
[回退建议内容]
```

## 输入输出

- 输入：待验证的论文文件
- 输出：验证报告 → `paper/review-report.md`（追加模式）

## 文件读写约定

| 操作 | 文件 |
|------|------|
| 读取 | `paper/metadata.json`, 待验证的 `paper/draft-*.md`, `references/checklists.md` |
| 写入 | `paper/review-report.md`（追加，带时间戳，不覆盖） |

## 注意事项

- 本 Skill 只做规范性检查，不做真实验证（无法上网核实文献和数据）
- 超出能力范围的检查必须标注"待人工核查"
- 验证报告使用追加模式，保留历史记录
- 如果 `paper/metadata.json` 不存在，提示用户先加载 `paper-coordinator`
